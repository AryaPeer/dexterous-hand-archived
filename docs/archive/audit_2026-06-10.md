## The bug (critical — peg task was unwinnable in reverse)

`get_insertion_depth_jax` measured depth of the peg's lowest point below the
hole-entrance **plane** with **no lateral containment**. That was a valid
insertion proxy while the entrance sat at table level; round-16 (c9dc70a)
raised the entrance to `table + 0.08` to fix the knuckle-cap descent ceiling,
which silently broke the proxy. Measured against the real compiled scene:

| peg pose                                  | depth  | fraction | success-depth met? |
|-------------------------------------------|--------|----------|--------------------|
| upright on table, 4cm from hole            | 0.0801 | **1.000**| yes |
| lying flat on table, 10cm away             | 0.0801 | **1.000**| yes |
| upright in table corner, 22cm away         | 0.0801 | **1.000**| yes |
| **genuinely inserted, on hole bottom**     | 0.0575 | **0.757**| yes |

A peg sitting anywhere on the table out-scored a fully inserted one. Chain of
consequences:

- The reward hold counter and the env's `insertion_complete` used this raw
  depth → **any peg at table level for 10 steps ended the episode as
  `is_success=1`**.
- In pre-grasped episodes (`p_pre_grasped=1.0` for the first ~12M steps of a
  full run) `was_lifted` can never latch (peg starts at grasp_site, only
  descends, never gains +5cm), so the drop penalty was structurally unable to
  fire → **opening the fingers was a free instant "success"** plus ~24/step of
  depth reward (`1−tanh(5·lateral)` = 0.80 at 4cm — useless as containment for
  a 1.2cm bore). Dominant policy from step 0 of training.
- **The 2026-06-01 5M sanity "PASS" was the exploit.** Its numbers
  (insertion_depth 0.060, axis_align 0.955, complete 3.24, stage 2.65,
  hold_steps 1.83) are what holding a peg low *near-but-not-in* the hole
  produces. The PEG_GATES floors derived from them would pass an exploiting
  run.
- **Both geometry guard tests validated the broken metric**:
  `test_peg_insertion_physically_reachable` descended the gripped peg at
  grasp_site (x≈−0.10, ~10cm lateral from the hole at (0,0)) — its "0.94
  achievable" was open-air descent next to the tube (tip ~8.6mm above the
  table). `test_success_depth_fits_in_tube` bounded by `hole_top_above_table`
  (0.08), but the true in-tube bound is the hole_bottom plate:
  `hole_depth − wt/2 = 0.0575`. At `hole_depth=0.05` the task is unwinnable
  yet both old tests passed.
- config.py's "achievable 0.94 / +0.24 margin" claim was the same artifact.
  **True in-tube ceiling: 0.757; margin over success_threshold 0.70 is
  +0.057 (~4.3mm) — thin but workable.**

## The fix (this working tree)

| file | change |
|---|---|
| `dexterous_hand/utils/mjx_helpers.py` | `get_insertion_depth_jax` takes a **required `bore_radius`** (= peg_radius + clearance) and returns 0 unless the capsule's lower-end axis point is laterally inside the bore. Gate lives **inside the helper** so no caller can bypass it — depth reward, hold counter, `complete`, `insertion_complete`, and the obs all flow through it. |
| `dexterous_hand/envs/peg_env.py` | Caches `_bore_radius` in `_rebuild_peg_caches`, which runs on every curriculum clearance change **before** the step/obs re-jit, so the tightening bore (4mm→1mm across stages) is re-baked correctly. Both call sites pass it. |
| `tests/test_geometry.py` | Rewritten. (1) `test_success_depth_fits_in_tube`: in-tube bound read from the **compiled model** (hole_bottom plate top → max 0.0575), not config arithmetic. (2) `test_insertion_depth_requires_lateral_containment` (new, fast): three on-table poses must measure depth exactly 0; bottomed-out in-tube peg must clear success depth. The direct regression test. (3) `test_peg_drop_insertion_reaches_success_depth` (slow): drops the peg into the actual tube, asserts settled fraction ≥ 0.73 and that it **holds** ≥ 0.70 for 50 more steps. The hand never needs table-level descent post-round-16 — it releases the peg over the bore. |
| `scripts/training/train_peg.py` | `PEG_GATES` rewritten — old depth floors (mean ≥ 0.040/0.050) were only reachable via the exploit and would auto-kill every honest run post-fix. Now: 10M = grip pose/progress (`axis_align ≥ 0.70`, `stage ≥ 1.5`, `peg_height ≥ 0.45`); 30M = insertion **exists** (`insertion_depth ≥ 0.001`, `hold_steps ≥ 0.05`, `axis_align ≥ 0.80`). Baselines are NaN until a clean sanity exists. |
| `dexterous_hand/config.py` | hole_top comment rewritten with true numbers (ceiling 0.757, margin +0.057, ceiling set by `hole_depth` not elevation); gate comment updated. |
| `dexterous_hand/rewards/peg_reward.py` | `lateral_gate_k` signature default 10.0 → 5.0 (now matches config; behavior unchanged — all callers passed it explicitly). |
| `assets/shadow_hand/runpod_peg_full.md` | §intro + §4 gate floors synced; contamination note added; "expected: 16.0" → 64.0 (also fixed in `runpod_grasp_full.md`). |
| `scripts/render_grasp_diagnostic.py` | Stale "mount_height=0.82" comment → 0.78. |

## Known-open design wart (flagged, deliberately not changed)

**Success-termination farming**: both tasks pay dense per-step shaping and
*terminate* on success with a one-shot/ramped bonus (250). Discounted-return
math says avoid success: grasp can farm ~23/step by holding lifted while
keeping cube speed just above ~0.11 m/s (`at_target` product < 0.85) ≈ 2,900
discounted vs ≈ 1,200 for the success path; peg (post-fix) can hover at
fraction ~0.69 for ~21/step indefinitely vs ~125 for completing. Options when
addressed: scale terminal bonus ≳ expected remaining-episode shaping, or stop
terminating on success. See memory note `peg_false_insertion_success`.
